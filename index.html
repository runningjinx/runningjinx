Git相关知识总结一

发表于 2020-07-12 更新于 2020-07-27

在平常的开发中，我们都需要使用 git。基本的命令行大家都很熟悉了。本文将介绍其他的 git 内容。包括以下三方面内容：HEAD、master 与 branch 的关系、push 的本质是什么、PR 和 MR 是什么。

HEAD、master 与 branch

HEAD 是当前 commit 的引用，指向当前工作目录所对应的 commit。每个仓库只有一个 HEAD, 每次我们提交时 HEAD 都会一动到最新的 commit。
在 Git 中，HEAD 是 git 中一个独特的引用。除此之外，git 还有一种引用，叫做 branch(分支)。master 是主分支，和其他分支有两点区别。一是新建的仓库中第一个 commit 会被 master 自动指向。二是 git clone 时，会自动 checkout 出 master。
通俗来说，我们可以把 branch 理解为从初始 commit 到 branch 所指向的 commit 之间所有 commit 的一个串。虽然 master 在开发中处于核心地位，但是所有的 branch 都是相等的。

当我们 push 的代码的时候，实际做了了什么？

大部分人会觉得 push 就是用把本地的代码提交到中央仓库去，用本地内容覆盖远端分支的内容。其实这个说法是不够准确的。实质上，push 所做的事情是：把当前 branhc 的位置（即它指向哪个 commit）上传到远端仓库，并把它的路径上的 commit 一起上传。需要注意的是，push 的时候上传当前分支并不会上传 HEAD，远程仓库的 HEAD 是永远指向默认分支（即 master）的。

同事经常说的的 PR 和 MR 是什么？

PR: pull request。MR: merge request。这两个并不是 git 的内容，而是 git 仓库提供方所提供的一种便捷功能。它可以让团队成员方便讨论一个 Branch，并在讨论结束后一剑合并 branch 到这个 master。例如，当我们在一个分支上完成我们的开发后。可以先把 branch push 到远程仓库去。之后，我们新建一个 pull request 或者 merge request。然后我们的同事就可以在 Github 上看到我们创建的 PR 或者 MR 了。在 PR 或者 MR 的页面上，同事可以 review 我们的代码，提出建议。我们可以根据他们的建议重新修改代码，push 上去。页面也会随着我们的 push 展示出最新的 commit。当你和同事达成一致，都认为可以合并了，可以点击 merge 的按钮，GitHub 就会自动帮我们合并两个分支的代码。目前来说，这种工作模式是商业项目开发中最为流行的工作流。

以上就是本篇博客的所有内容。下一期我会深入说一下 git 的一些高级用法。
